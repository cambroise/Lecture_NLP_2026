5.2.1 1) Data preparation and character-level tokenizer

Notebook cells: 1 – 8

Goal: Load a raw text corpus, build a character-level vocabulary, and encode the full dataset as a sequence of integers.

Key ideas

  • The tiny Shakespeare dataset contains 1,115,394 characters drawn from 65 unique characters (letters, punctuation, newlines).
  • A character-level tokenizer maps each character to an integer: encode turns a string into a list of integers, decode reverses it.
  • The full corpus is stored as a torch.Tensor of dtype torch.long.
  • The data is split 90 % train / 10 % validation before any model is built.

Exercises

  1. Print the vocabulary (all 65 characters). Which characters appear? Is there anything surprising?
  2. Encode the string "Hello GPT" and decode it back. Verify the round-trip.
  3. What would change if you used a word-level tokenizer instead? Estimate the new vocabulary size and the new sequence length for the same corpus.
  4. The split is done at position int(0.9 * len(data)). Why is it important not to shuffle the data before splitting?

5.2.2 2) Batch loading and the context window

Notebook cells: 9 – 12

Goal: Understand how a single (input, target) tensor pair encodes block_size training examples simultaneously.

Key ideas

  • block_size is the maximum context length (number of tokens fed to the model at once).
  • For a sequence of length block_size + 1, there are block_size overlapping (context, target) pairs: when the input is x[:t+1], the target is x[t+1].
  • get_batch samples batch_size random starting positions and stacks the corresponding windows into tensors (B, T).

Exercises

  1. Run get_batch('train') and print xb, yb. For the first sequence in the batch, manually list all (context, target) pairs.
  2. Why does get_batch sample starting positions from range(len(data) - block_size) rather than range(len(data))?
  3. Change block_size to 4 and batch_size to 2 and rerun. How does this affect the number of training examples per batch?

5.2.3 3) Bigram language model

Notebook cells: 13 – 16

Goal: Implement the simplest possible language model as a baseline before introducing attention.

Key ideas

  • A bigram model predicts the next character using only the current character: the token embedding table directly provides the logits for the next token.
  • The loss is cross-entropy between the predicted distribution and the true next character.
  • After 100 steps of AdamW (learning rate 1e-3, batch size 32), the loss drops from ≈ 4.88 to ≈ 4.66, and the model generates readable (but random) character sequences.
  • Generation uses multinomial sampling from the softmax distribution, appending one token at a time.

Exercises

  1. What is the cross-entropy loss of a uniform random model over 65 characters? Compare with the initial loss of 4.88.
  2. The generate method uses the whole sequence as context, but the bigram model only ever looks at the last token. Why is this not a problem here, but would be a problem for a Transformer?
  3. Train for 1000 steps instead of 100. Does the loss continue to decrease? Generate a sample and comment on its quality.
  4. Modify generate to use greedy decoding (argmax) instead of sampling. How does the output change?

5.2.4 4) The mathematical trick in self-attention

Notebook cells: 17 – 22

Goal: Understand weighted aggregation of past tokens via matrix multiplication, the foundation of self-attention.

Key ideas

  • A token at position \(t\) should only communicate with positions \(\leq t\) (causal constraint).
  • The simplest aggregation: \(x_t^{\text{bow}} = \frac{1}{t}\sum_{i \leq t} x_i\) (bag-of-words average).
  • Version 1 (loop): explicit for loop over positions — correct but slow.
  • Version 2 (matrix multiply): replace the loop with a lower-triangular matrix multiply. If wei is the \((T \times T)\) lower-triangular matrix normalised by row, then wei @ x computes all position averages in one operation.
  • Version 3 (softmax): initialise wei = zeros, fill future positions with -inf, then apply softmax — this is exactly the uniform-attention baseline.

Exercises

  1. Verify that versions 1, 2 and 3 produce identical outputs using torch.allclose.
  2. Print the weight matrix wei for version 2 (shape \(T \times T\)). What does each row represent?
  3. What happens if you remove the masking step in version 3? What values does softmax assign to future positions?
  4. In version 2, why must the rows be normalised (divided by their sum)? What would happen to the magnitude of xbow without normalisation?

5.2.5 5) Self-attention with Query, Key and Value

Notebook cells: 23 – 25

Goal: Replace the uniform average with data-dependent attention weights computed from learned projections.

Key ideas

  • Each position produces three vectors from the input \(x\): a query \(q = Qx\) (what am I looking for?), a key \(k = Kx\) (what do I contain?), and a value \(v = Vx\) (what do I communicate?).
  • Attention scores: \(\text{wei} = q k^T / \sqrt{d_k}\) — the dot product between the query at position \(t\) and the keys at all positions \(\leq t\).
  • After masking and softmax, the output is a weighted sum of values: \(\text{out} = \text{wei} \cdot v\).
  • Self-attention: \(Q\), \(K\), \(V\) are all produced from the same source \(x\).
  • Encoder vs decoder: removing the triangular mask gives an encoder block (all positions visible); keeping it gives a decoder block (causal, used for language modelling).

Exercises

  1. Print wei[0] (attention weights for the first sequence in the batch). Verify that the upper triangle is exactly 0. Which position attends most to which?
  2. The scaling factor is \(1/\sqrt{d_k}\) (head_size**-0.5). Cells 26–31 show why: compute wei.var() with and without the scaling. What happens to the softmax distribution when the variance is large?
  3. Remove the causal mask (delete the masked_fill line). The model can now look into the future. Why is this valid for an encoder but not for a language model?
  4. Change head_size to 1, then to 64. How does the shape of out change?

5.2.6 6) Scaled attention and numerical stability

Notebook cells: 26 – 31

Goal: Understand why the \(1/\sqrt{d_k}\) scaling prevents the softmax from saturating.

Key ideas

  • If \(q\) and \(k\) are unit-variance, then \(q \cdot k\) has variance \(d_k\) — its standard deviation grows with dimension.
  • Without scaling, the dot products become large in magnitude, softmax converges to a one-hot distribution (“peaky”), and gradients vanish.
  • With scaling by \(1/\sqrt{d_k}\), wei stays unit-variance regardless of head_size.

Exercises

  1. Run k.var(), q.var(), wei.var() before and after applying * head_size**-0.5. Confirm that scaling restores unit variance.
  2. Compare torch.softmax(tensor * 1, dim=-1) and torch.softmax(tensor * 8, dim=-1) for the vector [0.1, -0.2, 0.3, -0.2, 0.5]. Describe the difference.
  3. At initialisation, why do we want the attention weights to be approximately uniform (not peaky)?

5.2.7 7) Layer normalisation

Notebook cells: 32 – 34

Goal: Implement Layer Normalisation and understand how it differs from Batch Normalisation.

Key ideas

  • LayerNorm normalises each sample independently across its feature dimension (not across the batch).
  • For input \(x\) of shape \((B, C)\): \(\hat{x} = (x - \mu_x) / \sigma_x\), then \(\text{out} = \gamma \hat{x} + \beta\) with learned scale \(\gamma\) and shift \(\beta\).
  • After LayerNorm, x[0,:].mean() ≈ 0 and x[0,:].std() ≈ 1 for every sample.
  • In the modern Transformer (“Pre-LN”), LayerNorm is applied before attention and feed-forward (not after as in the original paper).

Exercises

  1. Run the LayerNorm1d example with x = torch.randn(32, 100). Verify that x[0,:].mean() and x[0,:].std() are approximately 0 and 1.
  2. Check x[:,0].mean() and x[:,0].std() (one feature across the batch). Why are these not normalised?
  3. What is the difference between this and nn.BatchNorm1d? In which training scenarios does LayerNorm have an advantage?

5.2.8 8) Full GPT implementation

Notebook cells: 36 – 37

Goal: Assemble all components into a complete character-level GPT and train it.

Architecture

Token embedding  (vocab_size → n_embd)
Position embedding (block_size → n_embd)
  └── N × Block:
        Pre-LN → MultiHeadAttention (n_head heads of size n_embd/n_head) → residual
        Pre-LN → FeedForward (n_embd → 4·n_embd → n_embd, ReLU) → residual
Final LayerNorm
Linear projection  (n_embd → vocab_size)

Hyperparameters (notebook defaults)

Parameter Value
batch_size 16
block_size 32
n_embd 64
n_head 4
n_layer 4
learning_rate 1e-3
max_iters 5000
dropout 0.0

Results: 0.21 M parameters — train loss 1.66 / val loss 1.82 after 5000 steps.

Exercises

  1. Count the parameters in each component (Head, MultiHeadAttention, FeedFoward, Block). Verify the total of 0.21 M.
  2. The Block applies LayerNorm before the sub-layer (x + self.sa(self.ln1(x))). Swap to Post-LN (self.ln1(x + self.sa(x))). Does training still converge?
  3. Set dropout = 0.2 and retrain. Does the gap between train loss and val loss decrease?
  4. Double n_embd to 128 and n_layer to 6. How many parameters does the model have now? What train/val loss do you achieve?
  5. Train on a different text corpus (a novel, your own text). How does the generated output change?
